REST API reference

The LicenPro API is an ASP.NET Core app routed under /api/... on your deployment host (for example https://your-host/api). Two authentication styles matter: Bearer JWT (dashboard users) for most management routes, and X-API-KEY (product API key) for selected anonymous SDK/server integration endpoints.

Product API key

Used by POST /api/Licenses/validate via ApiKeyAuthFilter. Send header X-API-KEY: <your product key> (or legacy Authorization: X-API-KEY <key>). Keys are hashed server-side; store the plaintext only on integrators’ servers.

Licenses

Controller route prefix: /api/Licenses (class LicensesController).

GET /api/Licenses

Paginated license list for the authenticated operator. Query: pageNumber, pageSize, search, status, type.

GET /api/Licenses/{id}

Single license by GUID.

POST /api/Licenses

Create license. Body: CreateLicenseDto — required name, type (enum string: Perpetual, Trial, Subscription, NodeLocked, Floating, Concurrent), issuedTo, softwareReleaseId; optional expirationDate, maxActiveUsersCount, hardwareId, subscription/trial duration fields, entitlementSetIds, notes.

POST /api/Licenses/validate

AllowAnonymous + X-API-KEY. multipart/form-data: licenseKey, validationParams (JSON string), optional licenseFile. Returns validity + status + type; maps hardware/revocation errors to HTTP 403/400 per implementation.

GET /api/Licenses/validate/{licenseKey}

Anonymous simple validation; optional hardwareId query.

GET /api/Licenses/status/{licenseKey}

Anonymous status payload for SDK-style polling.

POST /api/Licenses/refresh/{licenseKey}

Refresh license material / state (optional JSON body RefreshLicenseDto).

GET /api/Licenses/{licenseKey}/activations

Anonymous activation snapshot for a key.

GET /api/Licenses/{id}/download

Authorized download of generated license file bytes.

POST /api/Licenses/{id}/generate-file

Authorized (re)generation of license file.

POST /api/Licenses/{id}/revoke

Revoke license (operator).

Hardware binding management includes GET /api/Licenses/pending-bindings, POST .../confirm-binding, reject-binding, reset-binding, unbind-device, and transfer endpoints — see controller for full list.

Products (RSA keys)

Prefix: /api/Products.

  • POST /api/Products/{id}/keys/generate
  • GET /api/Products/{id}/keys/status
  • GET /api/Products/{id}/keys/public
  • GET /api/Products/{id}/keys/public/download

Security utilities

Prefix: /api/security.

  • POST /api/security/derive-key — body: licenseKey, saltBase64; returns derived key material for protected file flows.
  • Session listing/revocation for logged-in users under same controller (admin features vary by action).

Online vs offline at the API boundary

Online integrations call validate, status, refresh, and activation/session endpoints so the server remains authoritative for revocation, seat counts, and hardware binding. Offline use is implemented inside the customer app via RSA + signed license.bin (see .NET SDK ValidateOfflineAsync) and does not hit these routes unless you add your own telemetry.

Rate limits

Production deployments may apply throttling per API key or tenant. Treat validate endpoints as latency-sensitive and cache responsibly on your side.
