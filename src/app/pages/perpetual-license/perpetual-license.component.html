Perpetual license

A perpetual license grants the right to use the software without a recurring subscription term. You sell “buy once, use forever” (optionally with maintenance expiry), and choose online or offline enforcement at issuance.

Long-term entitlement

No renewal cycle unless you add an optional end date.

Online or offline

Perpetual online vs offline mode in the create wizard.

license.bin

Signed file for offline-capable deployments.

Business model

Perpetual fits classic software economics: customer pays once (or pays for a version), vendor delivers ongoing value through optional maintenance, support contracts, or upgrade SKUs.

  • One-time license fee — desktop tools, engineering software, perpetual ERP modules.
  • Perpetual + maintenance — optional expiry date for updates/support while use rights continue (policy choice).
  • Channel / OEM — issued-to org label; many seats under one commercial deal.
  • Post-trial conversion — evaluator upgrades from Trial to perpetual paid key.

Benefits

  • For vendors — simple SKU story; works offline for air-gapped customers; RSA-signed files reduce piracy vs raw keys alone.
  • For customers — predictable cost; offline mode avoids dependency on your uptime for daily work.
  • For support — online perpetual shows activations in dashboard; revoke compromised keys without redeploying entire product.

Logic of use

Online perpetual — app validates with LicenPro on startup (or periodically). Server records device activations and optional live sessions. Revocation and seat limits apply in real time.

Offline perpetual — app validates license.bin locally with embedded public key. No activation rows unless customer later goes online. Best for factories, labs, or regions with no stable internet.

Open perpetual (product/access pattern) — key may work for any user with the file; “issued to” can show as open access in UI.

Example lifecycle

Scenario: Engineering firm — 50 offline seats

Vendor sells 50 perpetual offline licenses to a plant. Operator issues 50 keys + files. Installers copy license.bin to each workstation. Apps never call the API during normal use. Annual support renewal is a separate subscription or manual expiry on maintenance-only keys if the vendor uses that pattern.

Scenario: SaaS-adjacent desktop — online perpetual

Customer buys perpetual with online mode. Each install activates a device; dashboard shows 12 active devices. IT revokes one machine from Activations tab after laptop loss. Next validation on that device fails without affecting other seats.

Lifecycle stages

  1. Issue — type Perpetual; pick online/offline; product + release.
  2. Distribute — key + license.bin.
  3. Activate — online: server activation; offline: local file import.
  4. Operate — monitor Activations/Sessions if online.
  5. Revoke / transfer — license Overview when contract ends or user changes.
  6. Expire — only if you set optional expiry (e.g. maintenance window).

Dashboard workflow

  1. Prerequisites: product, release, RSA keys (RSA keys).
  2. Licenses → Create → Perpetual → online or offline mode.
  3. Detail tabs: Overview; Activations / Sessions when online.

Compare with other models

  • vs Subscription — no automatic term renewal; no recurring expiry unless you set one.
  • vs Trial — perpetual is paid long-term; trial is evaluation-only.
  • vs Floating / Concurrent — perpetual is usually per-user or per-device, not a shared seat pool.
