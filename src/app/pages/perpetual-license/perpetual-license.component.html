Perpetual License

A Perpetual License is a multi-user license that never expires. It's a common licensing model for desktop applications - customers pay once and use the software forever. Perfect for traditional software sales with no recurring fees.

What is a Perpetual License?

A Perpetual License grants a customer permanent access to your software for one-time purchase licensing. Unlike subscriptions or trials, it doesn't expire and works completely offline by default. A single license can be used by multiple users/devices (based on your configured limits and usage policy) while still being issued to a customer identity.

Never Expires

License remains valid forever unless you set an optional expiration date

Multi User

One license can be used by multiple users/devices (as defined by your limits)

Works Offline

No internet required - validates locally using RSA signatures

Optional Online Mode

Enable heartbeat monitoring for usage tracking and analytics

How Multi-User Perpetual Licensing Works

Perpetual licensing is flexible: you sell a one-time license to a customer identity, then control how it can be used through limits and policy (for example, number of activations/devices, or seats). The license stays valid as long as it is not revoked and the usage stays within the configured limits.

Customer Identity

Keep a consistent customer identifier (name + email) for support, re-issuing, and audit trails.

Usage Limits

Define what “multi-user” means for your product: seats, devices/activations, or another measurable limit.

Compliance & Control

Revoke a license if needed, and optionally enable online validation for stronger enforcement and tracking.

Validation Flow Details

Step-by-step breakdown of what happens during license validation:

1
File Loading

SDK reads the license.bin file from the specified path. If the file doesn't exist or is corrupted, validation fails with InvalidFormat status.

2
Signature Verification

SDK verifies the RSA-SHA256 signature using the provided public key. If the signature doesn't match, validation fails with SignatureMismatch status. This ensures the license file hasn't been tampered with.

3
License Type Check

If ExpectedLicenseType is set, SDK verifies the license type matches. If it doesn't match, validation fails with WrongLicenseType status.

4
Credential Validation

SDK checks that UserName matches IssuedTo and LicenseKey matches the license key in the file. If either doesn't match, validation fails with CredentialsMismatch status.

5
Expiration Check

If the license has an expiration date, SDK checks if it's still valid. If expired, validation fails with Expired status.

6
Online Validation (Optional)

If ForceOnlineValidation = true, SDK connects to the dashboard API to check license status. This verifies the license hasn't been revoked and establishes a session for tracking.

7
Feature Loading

SDK loads assigned features from the license file and makes them available through FeatureManager. Your application can then check which features are enabled.

8
Session Heartbeat (Online Mode)

If online validation is enabled, SDK automatically sends heartbeat pings to the dashboard every 5 minutes. This keeps the session alive and updates the "Last Heartbeat" timestamp in the dashboard.

Offline vs Online Validation (Recommended Strategy)

Perpetual licenses work offline by default, which is ideal for desktop and secure environments. If you need stronger enforcement, analytics, or device management, enable online validation for periodic checks.

Use Offline Validation When

Customers operate without reliable internet, you ship air-gapped software, or you prioritize zero-runtime dependencies on your backend.

Enable Online Validation When

You need usage telemetry, proactive abuse prevention, rapid revocation, or a strict device/seat enforcement workflow.

Hybrid Approach

Validate offline on every launch, and periodically re-check online (for example, once per day/week) to keep UX smooth but maintain control.

Implementation Checklist

Use this checklist to ensure your perpetual license flow is robust and supportable:

Define Limits

Decide whether your multi-user policy is based on activations/devices, seats, or another metric, and configure it consistently in dashboard + app.

Protect Your Public Key

Ship the public key with your app and treat it as a required dependency for offline validation (never embed private keys).

License File Storage

Store the license file in a stable location and provide a UX for import/replace when customers reinstall or migrate devices.

Support Playbook

Document how to re-issue licenses, handle migration, and resolve exceeded-limits scenarios without blocking legitimate customers.

Common Issues & Troubleshooting

Validation Fails After Reinstall

Ensure the license file is re-imported and the application uses the same customer identity fields used when issuing the license.

Exceeded Limits

Provide a clear message explaining what limit was exceeded (devices/seats) and how to resolve it (deactivate old device, upgrade tier, or contact support).

Revoked License

Guide the customer to renew or request a new license. In-app, show a dedicated “License Status” screen with the last validation result.

Creating a Perpetual License

Follow these steps to create a perpetual license:

1
Navigate to Licenses

Go to Licenses → Create License from your dashboard.

2
Select License Type

Choose "Perpetual" from the license type dropdown.

3
Enter Customer Information

Add the customer name and email. Use this identity consistently for validation and customer support workflows.

4
Configure Usage Limits

Set the allowed usage for the license (for example, seats/devices/activations) to match your multi-user policy.

5
Set Expiration (Optional)

Leave blank for perpetual validity, or set a date for limited-time perpetual licenses (e.g., for major version upgrades).

6
Configure Features (Optional)

Assign specific features to enable or disable functionality based on the license tier.

7
Generate and Deliver

Create the license and download the .bin file. Send it to your customer via email or download link.

Complete Flow: Dashboard → SDK → User Machine

Understanding how perpetual licenses flow from creation in the dashboard to validation on the user's machine:

1
Dashboard: Create License

In the LicenPro Dashboard, navigate to Licenses → Create License. Select Perpetual type, enter the end user's email (e.g., user@example.com), select the product release, and configure optional settings like expiration date and features.

Dashboard Configuration
License Type: Perpetual
End User Email: user@example.com
Product: PhotoEditor Pro
Release: v1.0.0
Expiration: (Optional) 2027-12-31
Features: Pages Edit, Export PDF
Activation Mode: Offline (or Online for tracking)
2
Dashboard: Generate License Files

The dashboard generates three critical files that link the license to the user's machine:

  • license.bin - Encrypted binary file containing license data, signed with RSA private key
  • LicenseKey.txt - Plain text file with the license key (e.g., LP-A1B2-C3D4-E5F6)
  • PublicKey.pem - RSA public key for signature verification

These files are automatically downloaded as a ZIP package and can be sent to the end user via email or download link.

3
User Machine: Receive License Files

The end user receives the license package and extracts it to their machine. They should store these files in a secure location (e.g., C:\ProgramData\YourApp\License\).

4
SDK: Load License Files

Your application uses the LicenPro SDK to load and validate the license files. The SDK reads the license.bin file, verifies the RSA signature using the public key, and checks that the credentials match.

5
SDK: Validate Credentials

The SDK validates three key pieces of information:

  • UserName - Must match the email used when creating the license
  • LicenseKey - Must match the generated license key
  • PublicKey - Used to verify the license file signature

If all three match, the license is valid and the application can proceed.

6
SDK: Connect Session (Optional)

If online validation is enabled (ForceOnlineValidation = true), the SDK connects to the dashboard server to establish a session. This enables:

  • Real-time session tracking in the dashboard
  • Heartbeat monitoring (automatic ping every 5 minutes)
  • Usage analytics and feature tracking
  • Remote license revocation
7
Dashboard: Monitor Sessions

When online validation is enabled, you can monitor active sessions in the dashboard under Sessions. You'll see:

  • User email and device name
  • IP address and operating system
  • Connection time and last heartbeat
  • Online/Offline status

You can force disconnect sessions or revoke licenses remotely if needed.

Best Practices

User Email as Identifier: Use the customer's email as the UserName field when creating the license. Your app should validate using the same email string to ensure license belongs to the right user.
Graceful Expiration Handling: Even perpetual licenses can have optional expiration dates. Always check License.ExpirationDate and handle expired licenses gracefully with renewal prompts.
Feature-Based Tiers: Use features to create license tiers (Basic, Pro, Enterprise). This lets you sell different price points with the same codebase.

Validation Status Codes

Understanding validation status codes helps you provide better error messages to users:

Common Validation Status Codes

When validating a license, the SDK returns a status code indicating the result. These codes help you diagnose issues and provide meaningful feedback to users:

  • Valid — License is valid and active. All checks passed successfully.
  • InvalidFormat — License file is corrupted or has an invalid structure. The file may have been tampered with or incorrectly generated.
  • SignatureMismatch — RSA signature verification failed. This indicates the license was not signed with the correct private key or the public key in your app doesn't match.
  • CredentialsMismatch — The UserName or LicenseKey provided to the SDK doesn't match the values stored in the license file.
  • WrongLicenseType — The license type (Perpetual, Trial, Subscription, Floating) doesn't match what your application expects.
  • Expired — The license has passed its expiration date. Users need to renew their license.
  • Revoked — The license has been revoked by an administrator and is no longer valid.
  • DeviceBlocked — The current device is blocked from using this license, typically due to too many activations.
Status Code Reference
Valid                  - License is valid and active
InvalidFormat          - License file is corrupted or invalid
SignatureMismatch      - RSA signature verification failed
CredentialsMismatch    - UserName or LicenseKey don't match
WrongLicenseType       - License type doesn't match expected type
Expired                - License has expired
Revoked                - License has been revoked by admin
DeviceBlocked          - Device is blocked from using this license
CredentialsMismatch

Cause: UserName or LicenseKey in SDK doesn't match dashboard values.

Solution: Verify credentials exactly match (case-sensitive). Check for extra spaces or special characters.

SignatureMismatch

Cause: Public key doesn't match the private key used to sign the license.

Solution: Ensure you're using the correct PublicKey.pem file. Re-download license package if corrupted.

Session Disconnects

Cause: Network issues or heartbeat timeout.

Solution: Check internet connection, verify firewall allows connections to dashboard API, implement reconnection logic.
